# Grafana Dashboard Specification

## 1. Purpose

This document defines the initial Grafana dashboards for the observability platform.

The goal is to make the pilot immediately useful for operators and developers without creating noisy or fragile dashboards. The first dashboard set should answer these questions quickly:

- Is the service healthy right now?
- Are requests or jobs failing?
- Where is latency increasing?
- Is telemetry itself healthy?
- Can an engineer move from a metric spike to the relevant trace and logs?

This specification is intentionally dashboard-first and behavior-first. Exact metric names may vary slightly depending on OpenTelemetry metric translation, but the dashboard intent, filters, required labels, and PromQL shape should remain stable.

## 2. Dashboard Design Principles

- dashboards must focus on golden signals first
- every dashboard must support filtering by `service` and `deployment.environment`
- label usage must avoid high-cardinality dimensions
- raw path values with IDs must not be used as a grouping dimension; use normalized route templates
- dashboards should prefer a few decision-making panels over large grids of low-signal charts
- platform dashboards must include telemetry-pipeline health, not just application behavior
- logs and traces must be reachable from the same operational flow

## 3. Required Labels and Fields

The following dimensions should be consistently available across panels where applicable:

- `service`
- `deployment.environment`
- `instance`
- `route`
- `status_code`
- `job_name`
- `status`
- `level`
- `trace_id`
- `span_id`

Do not use these as metric labels:

- `user_id`
- `email`
- `request_id`
- raw URLs with embedded identifiers
- payload fields with unbounded values

## 4. Dashboard Set

### 4.1 Service Overview

Audience:

- backend engineers
- on-call responders
- engineering leads

Purpose:

- show the health of an API service in one view
- identify traffic shifts, errors, or latency regressions quickly
- support drilldown by route and environment

Recommended filters:

- `service`
- `deployment.environment`
- `instance`
- `route`

Panels:

#### Request Rate

Intent:

- show request throughput over time

Visualization:

- time series

Metric meaning:

- HTTP requests per second

PromQL candidate:

```promql
sum by (service) (
  rate(http_server_request_duration_seconds_count{service="$service", deployment_environment="$environment"}[5m])
)
```

Fallback pattern:

- use the request counter generated by OTel HTTP server instrumentation if the histogram count suffix differs

#### Error Rate

Intent:

- show the percentage and raw rate of failed requests

Visualization:

- time series plus stat

PromQL candidate:

```promql
(
  sum(rate(http_server_request_duration_seconds_count{service="$service", deployment_environment="$environment", status_code=~"5.."}[5m]))
/
  sum(rate(http_server_request_duration_seconds_count{service="$service", deployment_environment="$environment"}[5m]))
) * 100
```

#### Latency P50 / P95 / P99

Intent:

- show whether the service is slow overall and whether tail latency is rising

Visualization:

- time series

PromQL candidates:

```promql
histogram_quantile(0.50, sum by (le) (rate(http_server_request_duration_seconds_bucket{service="$service", deployment_environment="$environment"}[5m])))
```

```promql
histogram_quantile(0.95, sum by (le) (rate(http_server_request_duration_seconds_bucket{service="$service", deployment_environment="$environment"}[5m])))
```

```promql
histogram_quantile(0.99, sum by (le) (rate(http_server_request_duration_seconds_bucket{service="$service", deployment_environment="$environment"}[5m])))
```

#### In-Flight Requests

Intent:

- show current active request pressure

Visualization:

- stat or gauge

PromQL candidate:

```promql
sum(http_server_active_requests{service="$service", deployment_environment="$environment"})
```

#### Top Endpoints by Traffic

Intent:

- identify which routes drive most load

Visualization:

- bar chart

PromQL candidate:

```promql
topk(10,
  sum by (route) (
    rate(http_server_request_duration_seconds_count{service="$service", deployment_environment="$environment"}[5m])
  )
)
```

#### Top Endpoints by Latency

Intent:

- identify the slowest routes by tail latency

Visualization:

- bar chart

PromQL candidate:

```promql
topk(10,
  histogram_quantile(0.95,
    sum by (route, le) (
      rate(http_server_request_duration_seconds_bucket{service="$service", deployment_environment="$environment"}[5m])
    )
  )
)
```

#### Status Code Breakdown

Intent:

- visualize response-class distribution

Visualization:

- stacked time series or pie chart

PromQL candidate:

```promql
sum by (status_code) (
  rate(http_server_request_duration_seconds_count{service="$service", deployment_environment="$environment"}[5m])
)
```

#### Process Uptime

Intent:

- detect frequent restarts or unhealthy runtime behavior

Visualization:

- stat

PromQL candidate:

```promql
process_uptime_seconds{service="$service", deployment_environment="$environment"}
```

#### CPU Usage

Intent:

- show service CPU pressure

Visualization:

- time series

PromQL candidate:

```promql
rate(process_cpu_time_seconds_total{service="$service", deployment_environment="$environment"}[5m])
```

#### Memory Usage

Intent:

- show service memory growth and pressure

Visualization:

- time series

PromQL candidate:

```promql
process_resident_memory_bytes{service="$service", deployment_environment="$environment"}
```

### 4.2 Worker Overview

Audience:

- backend engineers
- job processing owners
- on-call responders

Purpose:

- measure job throughput, failures, and execution time
- identify unhealthy worker instances and bad job types

Recommended filters:

- `service`
- `deployment.environment`
- `instance`
- `job_name`

Panels:

#### Jobs Started Rate

```promql
sum by (service) (
  rate(worker_jobs_started_total{service="$service", deployment_environment="$environment"}[5m])
)
```

#### Jobs Completed Rate

```promql
sum by (service) (
  rate(worker_jobs_completed_total{service="$service", deployment_environment="$environment"}[5m])
)
```

#### Jobs Failed Rate

```promql
sum by (service) (
  rate(worker_jobs_failed_total{service="$service", deployment_environment="$environment"}[5m])
)
```

#### Job Success Percentage

```promql
(
  sum(rate(worker_jobs_completed_total{service="$service", deployment_environment="$environment"}[5m]))
/
  sum(rate(worker_jobs_started_total{service="$service", deployment_environment="$environment"}[5m]))
) * 100
```

#### Job Duration P50 / P95 / P99

```promql
histogram_quantile(0.50, sum by (le) (rate(worker_job_duration_seconds_bucket{service="$service", deployment_environment="$environment"}[5m])))
```

```promql
histogram_quantile(0.95, sum by (le) (rate(worker_job_duration_seconds_bucket{service="$service", deployment_environment="$environment"}[5m])))
```

```promql
histogram_quantile(0.99, sum by (le) (rate(worker_job_duration_seconds_bucket{service="$service", deployment_environment="$environment"}[5m])))
```

#### Active Jobs

```promql
sum(worker_jobs_active{service="$service", deployment_environment="$environment"})
```

If this metric is not implemented in the first release, keep the panel placeholder documented but disabled in the initial dashboard JSON.

#### Retry Count

```promql
sum by (job_name) (
  rate(worker_job_retries_total{service="$service", deployment_environment="$environment"}[5m])
)
```

If retry instrumentation is not implemented in the first release, treat this as a phase-two panel.

#### Worker CPU and Memory

Use the same process-level panels as the API dashboard, filtered to worker services.

#### Error Log Volume

Purpose:

- correlate worker failures with spikes in error logs

Data source:

- Loki

Suggested query shape:

- count log lines over time where `service="$service"` and severity is `error`

### 4.3 Observability Platform Health

Audience:

- platform owners
- on-call responders

Purpose:

- determine whether telemetry loss or delay is caused by the platform rather than the application

Recommended filters:

- `deployment.environment`
- `instance`
- collector component

Panels:

#### Local Collector Export Success / Failure

Metric meaning:

- successful and failed export attempts from local collectors

PromQL candidate:

```promql
sum by (instance) (
  rate(otelcol_exporter_sent_spans[5m])
)
```

Companion error panel:

```promql
sum by (instance) (
  rate(otelcol_exporter_send_failed_spans[5m])
)
```

Equivalent log and metric variants should be added for logs and metrics if exported separately.

#### Central Collector Export Success / Failure

Use the same collector metric family, filtered to the central collector instance and grouped by exporter.

#### Dropped Telemetry by Signal

PromQL candidates:

```promql
sum by (signal) (
  rate(otelcol_processor_dropped_items[5m])
)
```

If the collector exposes signal-specific names instead of a `signal` label, split this into dedicated panels for logs, metrics, and traces.

#### Collector Queue Depth

```promql
sum by (instance, exporter) (
  otelcol_exporter_queue_size
)
```

#### Collector CPU and Memory

```promql
process_resident_memory_bytes{service=~".*otel.*"}
```

```promql
rate(process_cpu_time_seconds_total{service=~".*otel.*"}[5m])
```

#### Prometheus Target Health

```promql
up
```

Filtered to collector and observability-stack scrape targets.

#### Loki / Tempo Ingestion Health

If backend self-metrics are enabled, add:

- ingestion rate
- error rate
- rejected samples or spans

If not available in phase one, keep these panels documented as optional platform maturity upgrades.

#### Central EC2 CPU, Memory, Disk Usage

These require host metrics, typically via node exporter or equivalent host-level metrics collection.

PromQL candidates:

```promql
100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
```

```promql
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100
```

```promql
(1 - (node_filesystem_avail_bytes{fstype!="tmpfs"} / node_filesystem_size_bytes{fstype!="tmpfs"})) * 100
```

If node-level metrics are not present in phase one, keep these panels marked as pending host-metrics enablement.

### 4.4 Logs and Traces Drilldown

Audience:

- developers debugging incidents
- on-call responders following spikes from other dashboards

Purpose:

- move from metrics to traces to logs without leaving Grafana

Recommended filters:

- `service`
- `deployment.environment`
- `trace_id`
- severity

Panels and views:

#### Logs by Service

Data source:

- Loki

Query shape:

- all logs for selected service and environment

#### Logs by Severity

Data source:

- Loki

Query shape:

- filter by `level`

#### Error Log Volume Over Time

Data source:

- Loki

Query shape:

- count error logs over time for selected service

#### Slow Trace Listing

Data source:

- Tempo

Criteria:

- traces above a selected duration threshold

#### Failed Trace Listing

Data source:

- Tempo

Criteria:

- traces with error status

#### Worker Failure Trace Listing

Data source:

- Tempo

Criteria:

- worker traces filtered by service and failure outcome

#### Trace-to-Log Correlation

Requirement:

- log lines must preserve `trace_id`
- Grafana links should allow pivot from a trace view to Loki results filtered by the same trace identifier

## 5. Minimum Metrics to Emit

The first dashboard version assumes the following metric categories exist:

- HTTP request count
- HTTP request duration histogram
- HTTP active requests
- worker jobs started count
- worker jobs completed count
- worker jobs failed count
- worker job duration histogram
- process CPU
- process memory
- process uptime
- collector sent telemetry
- collector failed exports
- collector queue size
- collector dropped items

## 6. Panels to Defer if Not Yet Instrumented

The following panels are useful but can be disabled initially if the signal does not exist yet:

- worker active jobs
- worker retry count
- node CPU, memory, disk
- Loki ingestion internals
- Tempo ingestion internals

These should remain in the spec even if the first JSON dashboard version omits them.

## 7. Alerting Relationship

Dashboards and alerts should complement each other:

- alerts should be defined in Prometheus rules
- Grafana dashboards should help explain why an alert fired
- dashboards must not become the only place critical thresholds are encoded

Initial alert-backed dashboard areas:

- high 5xx error rate
- p95 latency breach
- service down
- collector export failure
- central disk pressure

## 8. Validation Checklist

Before considering the dashboard specification ready for implementation, validate that:

- every panel maps to an expected telemetry source
- every dashboard supports service and environment filtering
- no required panel depends on high-cardinality labels
- API service panels are supported by planned Go API instrumentation
- worker panels are supported by planned Go worker instrumentation
- platform panels are supported by collector or host metrics
- trace-to-log correlation depends on fields already required by the logging standard

## 9. Recommended Next Artifacts

After this specification, the next concrete deliverables should be:

- Grafana dashboard JSON definitions
- Prometheus alert rules aligned to the dashboard panels
- datasource provisioning config
- runbook links from key dashboards
